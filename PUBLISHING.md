## Release limitations

Version 1.2.0 uses truthful implementation build `1.2.0-s1`, wire lineage `protocol-1.6-api-schema-12`, API schema 12, private store schema 13, and the frozen `1.1.8-cf50` legacy façade. Actual cf50 is the one ordinary live predecessor admitted through the digest-pinned closed path when no S1 admission offer exists; no semver/build-list expansion, protocol-only trust, or downgrade fallback broadens that path. cf50 adds fail-closed historical rollback supersession proof and exact removed-deterministic migration-shadow retirement without changing the mutation vocabulary or schema. cf49 closes installed manifest-path executable selection and adds a built-ins-only pre-import diagnostic bootstrap without changing the mutation vocabulary or schema-12 entity contract. cf48 adds exact delayed-winner waiting, deterministic startup barriers, and bounded atomic startup diagnostics without changing the mutation vocabulary or schema-12 entity contract. cf47 contributes forward-only pre-spend rollback projection recovery, nonfatal heartbeat-owned recovery blocking, and real-cf45 active-session socket-loss replacement certification on top of cf46 endpoint-first lifecycle attestation, exact socketless-owner retirement/replacement with durable-authority preservation, phase-accurate owned-saga recovery, deterministic worktree identity, and transactional audited duplicate-projection retirement. `1.1.0-cf42` / protocol-1.5 / schema-11 is an exact replay/migration predecessor and may be retired only by `upgrade-schema11` after global process authority is drained and a verified backup is recorded. Protocol-1.4/schema-10 build `1.1.0-cf41` is an explicit semantic-migration input: free-form EXCLUSIVE rows become visible legacy authority and are never blessed as new operations. Protocol-1.3/schema-9 builds (`1.0.1-cf38` through `1.0.3-cf40`) remain explicit historical migration inputs, not wire-compatible live brokers. Every operation re-attests the exact endpoint; unknown builds, protocol/schema drift, or an unavailable live lock fail loudly without replacement or signaling. Historical known-build upgrade intents remain read-only; exact build identity remains mandatory for migration and the `0.13.0-cf34` protocol-1.2/schema-6 upgrade transaction. Recovery commands do not retire a shared coordinator themselves—migration apply, verify, rollback, and cutover own drained retirement under the global migration operation lock. Certification must prove a second repository cannot mutate after writer authority/backup, all repositories participate in the pre-retirement drain, recovery attachments serialize on the same lock, and run-catalog activation remains paginated beyond 256 durable runs.

Automatic predecessor signaling is not an OS-handle-bound mathematical kill: Node leaves a documented identity-read-to-signal PID race. macOS certification requires `/usr/bin/python3` as the fail-closed libproc bridge; absence or incomplete ABI results forbid automatic retirement.

The separately certified opaque `tool_call_id` fix uses one shared receipt/terminal schema and runtime validator: no pipe interpretation or normalization, exact 1–200 Unicode-code-point bounds, and explicit NUL rejection.

Phases 28–35 provide the transactional coordinator, durable run/session/child supervision, end-to-end peer claim negotiation, offline mailbox replay, automatic terminal-evidence release-condition reconciliation, owner-scoped recoverable worktree/Git sagas with dirty-work preservation and foreign-run isolation, transactional edit-lease/change-reservation separation, persisted wait-for/deadlock resolution, starvation aging, contradiction-only escalation arbitration, and migration/rollback/cutover implementation. CF-9 release-certification scaffolding now emits matrix, scale, packed-migration, security, and SBOM evidence; only the release lead may mark certification complete after this checklist and independent final bughunt review.

Do not market the package as a fully autonomous live-provider scheduler. The current package supplies the Autopilot surfaces and offline/default QA needed for parent-prompt-driven orchestration, child runner validation, scheduler config/dispatch planning, sparse-by-default worktrees, checkout-profile snapshots, disk gates, claim-driven materialization, commit-bound observation expansion, per-unit worktrees, runtime-owned unit mergeback, validation staleness, claim GC, state-store persistence, context handoff, onboarding, runtime close/merge/abort, package distribution, Quality vNext spec/status gates, durable purpose state, worktree-scoped git guards, execution-audit generation/classification, child-created commit evidence, scope/protected-path adjudication helpers, work-item lifecycle/closure gates, and perfect-quality prompt doctrine. It does not ship compiled scheduling UI, PTY/TUI coverage, migration of unrelated pre-Autopilot/HLO runtime formats, default automated live-provider child runs, network push/PR creation, hosted PR automation, or a default live-provider child-execution scheduler.

Live child execution witnesses must be opt-in, use the exact fixed `openai-codex/gpt-5.6-sol|terra|luna` role assignment through Pi subscription authentication, avoid metered frontier API routes, run against a disposable workstream, and launch through `autopilot-agent-run` with a reviewed roster-compliant unit spec. The broader subscription-provider identity patterns remain compatibility support for parsing historical evidence, not launch authorization.
